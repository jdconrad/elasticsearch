# JavaScript test fixes and speculations

This file documents the approach taken for lang-javascript tests when migrating from Painless to JavaScript, and speculations for tests that still fail after substituting Painless syntax with valid JavaScript.

Most of this file was generated by Cursor AI. Responses from the developer are prefixed by DEV.

## Approach

- **Painless-only tests** (no JavaScript equivalent): Skipped with JUnit 4 `@Ignore("Painless-only: <reason>")`. Examples: primitive types (byte, short, char, int, long, float, double), `def`, Painless casts `(byte)`, `(int)`, etc., primitive arrays (`int[]`, `byte[]`, `boolean[]`), ClassCastException semantics for bitwise/compound assignment, Painless-specific APIs.
- **Tests with a JavaScript equivalent**: Script literals were converted (e.g. `int x = 5` → `let x = 5`, `5L` → `5` where acceptable, `def` → `let`, remove type casts). If such a test still fails, possible causes and fix ideas are noted below.

## Plan for next steps (by priority)

Ordered from most straightforward / lowest uncertainty to more speculative. Each step assumes re-running the lang-javascript test suite to verify.

### 1. FIX2 – Grammar: variable statement and bitwise `&` (highest priority)

**Goal:** Eliminate `reportAttemptingFullContext` so that `let x = 5; let y = 3; return x & y;` and `return 5 & 12;` parse without ambiguity.

**Actions:**

- **Variable statement (`let`):** The grammar already has `variableStatement` in the `statement` rule. The ambiguity at `d=2 (statement)` likely means another statement alternative (e.g. `expressionStatement`) can also start with the same tokens. Use ANTLR to disambiguate: ensure the lexer emits a distinct token for `let` (e.g. `Let` / `Var` / `Const`) so that only `variableStatement` can consume it, or reorder alternatives so `variableStatement` is tried before `expressionStatement`. If the lexer can treat `let` as an identifier in some contexts, remove that so `let` is always a keyword in script scope. Re-run tests and confirm no "reportAttemptingFullContext d=2 (statement), input='let'".
- **Bitwise `&`:** The grammar has both `BitAndExpression` (`'&'`) and `LogicalAndExpression` (`'&&'`). The ambiguity at `d=99 (singleExpression), input='&'` suggests the parser can't decide between the two. Ensure the lexer produces two distinct tokens (e.g. `BitAnd` for `&` and `LogicalAnd` for `&&`) and that the lexer matches `&&` before `&` so a single `&` is never tokenized as two tokens. If the grammar uses the same token for both, split into two token types. Optionally use ANTLR's precedence/left-factoring so bitwise AND is tried before logical AND where appropriate. Re-run tests and confirm no "reportAttemptingFullContext … input='&'".

**Verification:** Run `./gradlew :modules:lang-javascript:test --tests "org.elasticsearch.javascript.*"` and check that AndTests.testBasics, testInt, testIntConst, testLong, testLongConst and AdditionTests.testBasics, testIntConst no longer fail with parser ambiguity.

### 2. FIX1 – Script-only fixes for "no viable alternative"

**Goal:** Where the failure is "no viable alternative at input 'x'" (or similar) and the **language** is not to be extended, change the **test script** to valid JavaScript that the current grammar already accepts.

**Actions:** Grep test output (or code) for "no viable alternative" and for each failing test script, rewrite the script to equivalent valid JS (e.g. avoid constructs that the grammar doesn't support, use supported statement/expression forms). Do not add new grammar rules for Painless-only or non-JS constructs.

**Verification:** Affected tests pass or fail for a different (non-parse) reason.

### 3. FIX5 – Walker: throw ScriptException on parse error

**Goal:** When the parser fails, tests that expect `ScriptException` (e.g. `expectScriptThrows`) should get a `ScriptException` instead of an `AssertionError` from the Walker's error listener.

**Actions:** In the Walker's error listener (the `BaseErrorListener` that currently throws `AssertionError` in `syntaxError`), replace that with creating and throwing a `ScriptException` (or wrapping the parse error in one) with an appropriate message and cause, so that script-execution and error-handling tests see the same exception type as in Painless. Prefer changing the Walker only, not the tests.

**Verification:** Tests that expect `ScriptException` for invalid scripts no longer fail with "Expected ScriptException but got AssertionError".

Compare with Painless's location recording and exception reporting and align.

**Verification:** Tests that assert script stack or column (e.g. `assertScriptElementColumn`, `assertScriptStack`) pass or are fixed with minimal test changes.
